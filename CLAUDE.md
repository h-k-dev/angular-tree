# angular-tree

High-performance standalone Angular tree component (react-arborist internals, Angular Material DX). Angular ≥ 21, zoneless, signals-only, CDK as the only runtime dependency.

## Session protocol — do this first, every session

1. Read **ROADMAP.md** — the single source of truth for phases, locked decisions, settled brainstorming, and open questions. Resume work from the current phase noted there. Post-v1 planning lives in **ROADMAP2.md** (nothing there starts before Phase 8 completes).
2. Read **STYLE.md** — mandatory coding style. All code (including stubs and specs) must pass its Review Checklist.

## Rules

- **Locked decisions in ROADMAP.md are not renegotiated silently.** If implementation reveals a problem with one, raise it explicitly before deviating.
- **When a decision gets settled in discussion, write it into ROADMAP.md immediately** (settled list, decisions table, or non-goals) — the roadmap must never lag the conversation.
- Public API changes go through the "Public API Sketch" section of ROADMAP.md before code.
- DX and Material familiarity outrank micro-optimization at the API boundary; performance work stays internal (flat model, virtualization).
- The tree ships no UI it doesn't own: no built-in menu, editor, or checkbox — directives + template contracts instead.
- **Scaffold with the Angular CLI** (`ng generate service|directive|component …`), never by hand — current Angular defaults apply (no `.service.ts`/`.directive.ts` file suffixes).
- **Modern Angular naming: no `Service`/`Directive`/`Component` class suffixes.** Names describe what the thing _does_ (`TreeController`, `TreeNodeToggle`, `AngularTree`) — see STYLE.md § Naming.

(Parking it BEHIND the opaque canvas was tried and rejected: occluded is *perfectly* invisible, not "almost".) `pointer-events: none` stops the ghost stealing clicks; proximity still fires as pointermove bubbles from the canvas. Pointer PROXIMITY (`data-near`, distance to the leading edge — not hover) **or `:focus-within`** expands it 320 → 460px, full opacity, deeper frost. **Keyboard reveal is CSS, not a handler — a real bug, not a preference**: the ROVING TABINDEX (`focusMode: 'roving'` — one row holds `tabindex="0"`) focuses a row, an opacity-faded element is STILL focusable, and the first cut's `(focusin)="near.set(true)"` got clobbered by every pointermove (which rewrites `near`), hiding the panel out from under the focused row. a signal can't be clobbered — and the selector is **`:has(:focus-visible)`, NOT `:focus-within`**: a roving tabindex takes focus on plain CLICK too, so `:focus-within` pinned the panel open forever after any click ("it stops sinking to the bg"); `:focus-visible` is the browser's own keyboard-modality bit, so mouse stays proximity-governed while keyboard pins. (Verification trap: synthetic `KeyboardEvent`s do NOT set input modality, so `:focus-visible` never matches under dispatched events — the keyboard path looks broken until driven with a REAL key press.) **`selectionMode: 'follow'`**: the default `'explicit'` moved FOCUS only, so arrows just scrolled the list past a canvas that never changed (the viewer reads `selectedKeys`, which arrows never wrote); under `'follow'`, `focusStep` → `#followFocus` replace-selects, so each arrow shows the next artwork — and since `#followFocus` early-returns on non-selectable nodes, arrowing across a DEPARTMENT keeps the last artwork instead of blanking. A transitioned property reads as "rule not applying" even when `matches()` is true; `transition: none` reveals the end state. Verify animated states by neutralising transitions and asserting the END state.** Browser-verified finding: **NO `loading="lazy"` on virtualized rows** — native lazy-loading reads the img's attach-time position, but a CDK virtual row is attached THEN `translateY`'d into place, so it captures a pre-transform offscreen position and never loads (in-viewport imgs stuck `complete:false`; eager `new Image()` of the same URL loaded in ~150ms); virtualization already IS the deferral, so lazy is both broken and redundant here — removed, documented inline.
